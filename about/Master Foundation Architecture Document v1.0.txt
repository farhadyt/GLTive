2. Platformanın missiyası

GLTive platformasının məqsədi şirkətlər üçün aşağıdakı capability-ləri vahid sistemdə birləşdirməkdir:

IT asset idarəetməsi
ticketing və service operations
infrastructure və network visibility
monitoring və event intelligence
security / SOC əsasları
AI-assisted operations
automation və self-healing
backup orchestration və recovery control
vendor-controlled on-prem productization
multi-tenant commercial packaging

Platforma bazara “sadə helpdesk” kimi yox, AI-driven enterprise IT operations platform kimi çıxmalıdır.

3. Əsas məhsul prinsipləri
3.1. Capability-first design

Platforma ekranlara görə yox, capability-lərə görə qurulacaq.

3.2. Multi-tenant by design

Tenant (müştəri şirkət) ayrımı bütün sistemin bünövrəsidir.

3.3. Productized on-prem architecture

Sistem yalnız daxili istifadə üçün yox, lokal mühitə quraşdırılıb satılan kommersiya məhsulu kimi qurulacaq.

3.4. AI as controlled infrastructure

AI sadəcə chat deyil; policy, audit, routing və approval ilə idarə olunan capability olacaq.

3.5. Security and audit first

Security, audit, licensing, activation və admin separation ilk gündən nəzərə alınacaq.

3.6. Extensible without rebuild

Sonradan yeni modul və yeni capability əlavə etmək sistemin skeletini dağıtmamalıdır.

3.7. Core data stays with customer

Müştərinin operational datası lokal mühitdə qalacaq. Vendor tərəfə yalnız licensing/commercial control metadata çıxacaq.

10. Pricing modeli
10.1. Əsas prinsip

Eyni modulu alan iki şirkət eyni qiyməti ödəməməlidir, əgər resurs və əməliyyat yükü fərqlidirsə.

10.2. Pricing formulu

Qiymət = Base Platform Fee + Activated Module Fee + Managed Resource Units + Site/Instance Factor + Optional Premium AI/SOC/Advanced Modules

10.3. Billable resource kateqoriyaları
endpoint count
server count
network device count
user/admin count
AI request quota
automation runs
integration connector count
storage/backup scope
site / instance sayı
10.4. Auto-discovery pricing

Platforma ilkin discovery ilə quote suggestion verə bilər.
Amma final qiyməti vendor admin təsdiqləməlidir.

10.5. Growth-friendly pricing

Hard tier model yox, smooth growth band modeli istifadə olunmalıdır ki, müştəri 50-dən 51-ə keçəndə şok sıçrayış olmasın.

18. Əsas risklər və boşluqlar
18.1. Scope overload

Çox capability-ni eyni anda yazmağa çalışmaq layihəni dondura bilər.

18.2. Phase və capability xəritəsini qarışdırmaq

Capability map memarlıq üçündür, icra siyahısı deyil.

18.3. Licensing və activation zəifliyi

Commercial control itirilərsə məhsul modeli dağılır.

18.4. AI overreach

AI imkanları controlsuz açılarsa təhlükəsizlik və etibar zərər görər.

18.5. Backup storage trap

Platformanın özü nəhəng native backup storage məhsuluna çevrilərsə əsas məhsul fokusdan çıxar.

18.6. Email security-də scope partlayışı

İlk mərhələdə tam mail provider qurmağa çalışmaq strateji gecikmə yaradar.

18.7. Pricing injustice

Resource, site və complexity nəzərə alınmasa qiymət modeli ya satışı öldürər, ya da mənfəəti.
